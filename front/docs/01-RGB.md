# AI가 코드를 짜는 시대, 삼권분립 방식으로 코딩해야 한다

### RGB 개발방법론 (Rules, Guard, Build)

“원하는 것을 바라기보다, 그 위에 있어라.”, *닐 도날드 월시, 신과 나눈 이야기 1권에서*

## 핵심 구조

| 현실의 삼권분립 | RGB 개발방법론 | 실제 형태 |
|---|---|---|
| 입법부 | Rules | 헌법 / 법률 / 스펙 |
| 사법부 | Guard | Test Cases |
| 행정부 | Build | Application Code |

---

## 요약

- AI 시대의 프로젝트는 단순히 코드를 빨리 만드는 것보다, **프로젝트의 질서를 잘 세우고 유지하는 것**이 더 중요해졌다.
- RGB 개발방법론은 그 질서를 **입법·사법·행정**처럼 나눠서 다루는 방식이다.
- **Rules**는 입법이다. 그리고 Rules는 다시 **프로젝트의 상위 구조 원칙을 정의하는 헌법**, **그 원칙을 실제 운영 규칙으로 내린 법률**, **구현할 기능을 정의하는 스펙**으로 나뉜다.
- **Guard(Test Cases)** 는 사법이다. Rules를 실제로 어겼는지 판정하고, 프로젝트의 질서를 테스트로 집행한다.
- **Build(Application Code)** 는 행정이다. 실제 서비스 코드, 컨트롤러, 유스케이스, 이벤트 핸들러 같은 애플리케이션 코드는 Rules와 Guard 안에서만 움직인다.
- 이 문서에서 말하는 “과거의 성공한 프로젝트”는 전혀 상업적 성공이 아니다. 내가 말하는 성공은 오직 **아키텍처적 성공**이다.
- 즉 경계가 좋았고, 테스트가 강했고, 기능을 계속 붙여도 쉽게 무너지지 않았던 구조적 성공을 뜻한다.
- **헌법과 법률은 도메인 로직이 아니라 구조(아키텍처) 규칙이다.**
- **스펙은 도메인 로직과 구현할 기능을 정의한다.**
- RGB는 한 번 선언하고 끝나는 문서 작업이 아니다. **Rules → Guard → Build → Rules 보강으로 이어지는 반복 루프**다.
- 먼저 Rules를 세우고, 그걸 Guard(Test Cases)로 집행 가능하게 만들고, 그 안에서 Build(Application Code)를 만든다. 그리고 구현 과정에서 드러난 문제를 바탕으로 다시 Rules와 Guard를 보강한다.
- 이 과정 전반을 가능한 한 AI가 최대한 수행하고, 사람은 전체를 검토하고 승인한다.
- 결국 RGB는 **AI가 코드를 많이 짜도 프로젝트가 무너지지 않게 만드는 반복 가능한 질서**를 만드는 방법이다.

---

새 프로젝트를 시작할 때, 클로드 코드나 GPT에게 이런 식으로 말해본 적이 한 번쯤은 있을 것이다.

- “A 프로젝트 스타일로 초기 세팅 잡아줘.”
- “컨트롤러는 B 프로젝트 방식으로 맞춰줘.”
- “테스트는 C 프로젝트 스타일을 참고해.”
- “패키지 구조도 예전에 잘 됐던 그 방식으로 가자.”

이건 이상한 일이 아니다. 오히려 너무 자연스럽다.

개발자는 새 프로젝트를 시작할 때 완전한 백지에서 출발하고 싶어 하지 않는다. 예전에 구조가 잘 잡혔던 프로젝트가 하나라도 있었다면, 그 프로젝트의 질서를 다음 프로젝트에서도 최대한 이어가고 싶어 한다. 패키지 경계, 테스트 스타일, 동기·비동기 흐름, 공통 코드의 위치, 실시간 채널의 역할 같은 것들이 한 번 잘 작동한 경험이 있다면, 다시 그 감각을 가져오고 싶은 건 당연하다.

문제는 여기서부터 시작된다.

그 질서가 정말 좋았다고 모두가 느끼고 있었는데, 막상 다음 프로젝트로 넘어오면 다시 비슷한 토론을 처음부터 반복하게 된다. 어디까지를 shared로 둘지, 어느 시점에 이벤트로 넘길지, 다른 도메인을 직접 참조해도 되는지, 테스트를 어디까지 통합으로 끌고 갈지, 실시간 채널에서 어디까지 허용할지 같은 이야기들이 또 나온다.

결국 예전에 분명히 잘해봤던 것인데도, 새 프로젝트에서는 다시 흐려진다.

왜 그럴까.

내 생각에는 이유가 단순하다.

좋았던 구조가 있었던 것은 맞는데, 그 구조가 **재사용 가능한 질서**로 고정되어 있지 않았기 때문이다.

- 사람은 기억한다. 하지만 기억은 기준이 되지 않는다.
- 팀은 감각을 공유한다. 하지만 감각은 테스트가 아니다.
- AI는 예시를 따라 한다. 하지만 예시는 원칙이 아니다.

그래서 나는 요즘 개발에서 중요한 것이 코드 생산량 그 자체가 아니라, **먼저 질서를 세우는 일**이라고 생각하게 됐다.

그리고 그 질서를 정리하는 가장 단순한 틀로 **RGB 개발방법론**이라는 이름을 붙였다.

RGB는 이렇게 읽으면 된다.

- **Rules**: 프로젝트의 질서를 정한다.
- **Guard**: 그 질서를 지키게 만든다.
- **Build**: 그 질서 안에서 구현한다.

조금 더 재밌게 말하면, RGB는 개발의 **삼권분립**이다.

- 입법은 Rules다.
- 사법은 Guard다.
- 행정은 Build다.

이 비유가 웃기게 들릴 수는 있다. 그런데 생각보다 잘 맞는다.

---

# 왜 이런 생각을 하게 됐나

AI 시대의 개발자가 원하는 것은 사실 복잡하지 않다.

- 가능하면 LLM이 많은 구현을 해줬으면 좋겠다.
- 기능 추가는 빨랐으면 좋겠다.
- 테스트는 충분했으면 좋겠다.
- 코드 품질도 유지됐으면 좋겠다.
- 그리고 기능을 계속 붙여도 프로젝트가 철학과 원칙을 잃고 무너지지 않았으면 좋겠다.

여기에 하나가 더 있다.

- 한 번 잘 만들었던 구조를, 다음 프로젝트에서도 다시 쓰고 싶다.

여기서 말하는 “잘 만들었던 구조”는 매출이 잘 나왔다는 뜻이 아니다. 사용자가 많았다는 뜻도 아니다. 투자받았다는 뜻도 아니다.

내가 말하는 성공은 **아키텍처적 성공**이다.

- 경계가 잘 나뉘어 있었다.
- 의존성이 건강했다.
- 공통 코드가 덜 오염됐다.
- 테스트가 구조를 지켜줬다.
- 기능을 계속 붙여도 프로젝트가 쉽게 무너지지 않았다.
- AI가 많은 구현을 해도 코드 퀄리티가 크게 흔들리지 않았다.

# “그 상식을 누가 모르나요? LLM도 그걸 알아요.”

- “테스트는 독립적으로 돌아가야지 ㅋㅋ”
- “도메인끼리 직접 물리면 나중에 꼬이지.”
- “공통 코드는 shared로 빼야지.”
- “비동기 흐름은 이벤트로 가는 게 낫지.”
- “SSE에서 상태 바꾸면 좀 이상하지 않나?”

이런 말들은 개발자에게 너무 익숙하다.

그런데 문제는 이것이다.

- 상식은 보통 문서로 남지 않는다.
- 문서로 남지 않으면 기준이 되지 않는다.
- 기준이 없으면 바쁠 때 타협된다.
- 타협이 반복되면 예외가 쌓인다.
- 예외가 쌓이면 구조가 무너진다.

그래서 필요한 것은

**좋은 생각이 아니라, 고정된 규칙이다.**

RGB에서 그게 바로 Rules다.

---

# 헌법과 법률은 도메인 로직을 몰라야 한다

더 정확하게 말하면 이렇다.

- **헌법과 법률은 도메인 로직을 모른다. 아키텍처를 알고 있다.**
- **스펙은 도메인 로직을 정의한다.**

즉 RGB에서 Rules는 하나의 레이어가 아닌 세개의 레이어로 구성된다.

- **헌법** : 구조 철학
- **법률** : 구조 규칙
- **스펙** : 도메인 로직과 구현할 기능

---

# Rules 단계는 왜 헌법·법률·스펙으로 나뉘어야 하나

| 구분 | 역할 | 성격 | 예시 |
|---|---|---|---|
| 헌법 | 상위 구조 원칙 | 추상적 | **도메인 간 직접 의존 금지** |
| 법률 | 실제 운영 규칙 | 구체적 | **모든 JPA 관련 테스트는 최종적으로 롤백** |
| 스펙 | 구현 대상 정의 | 기능 중심 | **회원 가입에서는 닉네임 입력이 필수이다.** |

헌법은 방향을 정하고,  
법률은 실행 규칙을 정하고,  
스펙은 구현 대상을 정한다.

---

# Guard는 사법부다

Guard는 단순한 테스트 묶음이 아니다.

**프로젝트의 사법부다.**

- 다른 도메인을 직접 참조했는가
- 읽기 전용 채널이 상태를 바꾸는가
- 테스트가 더 이상 독립적으로 돌지 않는가

이 규칙을 어기면 **테스트는 실패해야 한다.**

---

# Build는 행정부다

Build는 실제 구현이다.

컨트롤러, 유스케이스, 서비스, 이벤트 핸들러 같은 실제 코드는  
**Rules와 Guard 안에서만 움직인다.**

그래서 Build는 단순한 코딩이 아니라

**Rules와 Guard 아래에서 집행되는 행정이다.**

---

# 실전 방법론 : RGB는 반복된다

Rules (헌번/법률/스펙 추가/개선)
↓
Guard(Test Case 추가/개선)
↓
Build(구현)
↓
Rules 보강
↓
Guard 보강
↓
반복

---

# RGB는 사람이 하는 절차가 아니다

대부분을 **AI가 수행한다.**

- AI가 Rules 초안을 만든다
- AI가 Guard를 만든다
- AI가 Build를 한다
- AI가 Rules를 다시 개선한다

사람은 가끔 **검토하고 승인한다.**

---

# 마지막으로

RGB 코딩방식은 전혀 새롭지 않다.

좋은 팀은 원래

- 구조를 신경 썼고
- 테스트를 신경 썼고
- 경계를 신경 썼다.

달라진 것은 이것이다.

**이제 그 질서를 AI까지 포함해서 운영해야 한다.**

예전에는 좋은 규칙이 있으면 팀이 덜 망가졌다.  
지금은 좋은 규칙이 있으면 **AI가 코드를 덜 망가뜨린다.**

---

# 결론

- RGB 개발방법론은 AI 시대의 소프트웨어 개발에서 먼저 프로젝트의 질서를 세우고 그 질서를 반복적으로 발전시키기 위한 접근이다.
- Rules 단계에서는 헌법·법률·스펙을 정의/개선하여 프로젝트의 구조 질서와 구현할 기능을 정리한다.
- Guard 단계에서는 Test Cases를 통해 그 질서가 실제로 지켜지는지 판정하고 집행 가능하게 만든다.
- Build 단계에서는 그 질서 안에서 실제 Application Code를 구현한다.

이 반복 루프를 통해 **스펙의 깔끔한 구현과 아키텍처의 견고함을 지속적으로 유지하고 발전시킨다.**

## 추가 1 : 짧은 텍스트의 힘

복잡한 AI코딩 전략보다 **짧은 규칙**이 더 강하다.

- 헌법은 짧고 명확하다.
- 법률은 짧고 명확하다.
- 테스트는 명확하다.

LLM도 마찬가지다.

단순하고 명확한 규칙을 훨씬 잘 따른다.

## 추가 2 : 좋은 질서를 바라지 말고, 좋은 질서를 가져라. 

“원하는 것을 바라기보다, 그 위에 있어라.”, *닐 도날드 월시, 신과 나눈 이야기 1권에서*

RGB는 말한다.

“좋은 질서를 바라지 말고, 좋은 질서를 가져라. 그 질서 위에서 코드와 구조는 스스로 정렬된다.”
